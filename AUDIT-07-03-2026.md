## 🔴 PROBLEMAS CRÍTICOS — Bloquean funcionalidad

### BUG-V4-001 — `user_artifacts` definida dos veces (conflicto de tablas)

**Problema:**
- `src/core/rpg_models.py` define `UserArtifact` con `__tablename__ = "user_artifacts"`
- Archivos generados (`dice_system.py`, `gamification_updated.py`) crean tabla `user_artifacts` con SQL raw y esquema diferente (sin `equipped`, sin `uses_left`)

**Impacto:**
```python
OperationalError: table user_artifacts schema mismatch
```

**Corrección:**
Usar el modelo ORM existente en lugar de SQL raw:

```python
# En vez de INSERT raw SQL:
from src.core.rpg_models import UserArtifact

artifact_record = UserArtifact(
    user_id=user_id,
    artifact_id=artifact["id"],
    uses_left=1
)
db.add(artifact_record)
```

**Prioridad:** 🔴 CRÍTICA

---

### BUG-V4-002 — `dice_system.py` importa de `gamification.py` que no existe

**Problema:**
```python
from gamification import add_coins, add_wyrd, add_xp
```

En V4, estas funciones son métodos de `RPGService` en `src/services/rpg.py`. No existe módulo `gamification.py` en raíz.

**Impacto:**
```python
ModuleNotFoundError: No module named 'gamification'
```

Funciones afectadas:
- `daily_dice_roll()`
- `apply_combo_rewards()`
- Sistema de indulgencias completo

**Corrección:**
Reescribir usando `RPGService`:

```python
from src.services.rpg import RPGService

def apply_combo_rewards(db: Session, user_id: int, combo, streak_multiplier=1.0):
    rpg = RPGService(db, user_id)
    
    if rewards["xp"] > 0:
        rpg.add_xp(rewards["xp"], f"Combo: {combo.value}")
    
    if rewards["obolos"] > 0:
        rpg.add_coins("obolo", rewards["obolos"], f"Combo: {combo.value}")
```

**Prioridad:** 🔴 CRÍTICA

### BUG-V4-003 — `RPGService.add_coins()` solo acepta singular, pero `COIN_REWARDS` usa plural

**Problema:**

`src/services/rpg.py` → `add_coins()`:
```python
if coin_type == "obolo":      # ← singular
    profile.obolos += amount
elif coin_type == "dracma":   # ← singular
    profile.dracmas += amount
```

Pero `constants_new.py` → `COIN_REWARDS`:
```python
"habit_completed": {"obolos": 1}   # ← plural
```

**Impacto:**
Cuando se intenta aplicar recompensas, el `if` no matchea y **las monedas no se añaden silenciosamente** (retorna `False` pero sin error).

**Corrección:**
Añadir normalización plural→singular:

```python
_NORMALIZE = {
    "obolos": "obolo",
    "dracmas": "dracma",
    "tetradracmas": "tetradracma",
    "decadracmas": "decadracma"
}

def add_coins(self, coin_type: str, amount: int, reason: str = "") -> bool:
    coin_type = _NORMALIZE.get(coin_type, coin_type)  # ← añadir esta línea
    # ... resto del código
```

**Prioridad:** 🔴 CRÍTICA

### BUG-V4-004 — `indulgencias_log` y `dice_roll_log` no están en ningún modelo ORM

**Problema:**
El plan de implementación indica crear estas tablas con SQL raw. Pero V4 usa SQLAlchemy ORM con `Base.metadata.create_all()`. Las tablas raw **no se crearán automáticamente** con `init_db.py`.

**Impacto:**
```python
OperationalError: no such table: indulgencias_log
```

**Corrección:**
Añadir modelos ORM en `src/core/rpg_models.py`:

```python
class IndulgenciaLog(Base):
    __tablename__ = "indulgencias_log"
    
    id             = Column(Integer, primary_key=True, autoincrement=True)
    user_id        = Column(Integer, nullable=False)
    habit_type     = Column(String, nullable=False)
    type           = Column(String, nullable=False)   # prevention | absolution
    cost_json      = Column(Text)
    wyrd_recovered = Column(Integer, default=0)
    created_at     = Column(DateTime, default=datetime.utcnow)

class DiceRollLog(Base):
    __tablename__ = "dice_roll_log"
    
    id          = Column(Integer, primary_key=True, autoincrement=True)
    user_id     = Column(Integer, nullable=False)
    dice_count  = Column(Integer)
    roll_result = Column(Text)    # JSON
    combo       = Column(String)
    rewards_json= Column(Text)    # JSON
    artifact_id = Column(String)
    created_at  = Column(DateTime, default=datetime.utcnow)
```

**Prioridad:** 🔴 CRÍTICA

### ISSUE-V4-002 — Fórmula de nivel inconsistente

**Problema:**

`src/services/rpg.py`:
```python
new_level = int((profile.xp / 100) ** 0.5)   # fórmula sqrt continua
```

`constants_new.py`:
```python
LEVEL_XP = {1: 0, 2: 100, 3: 200, 4: 350, 5: 500, ...}   # tabla discreta
```

**Impacto:**
- Con fórmula sqrt: nivel 10 = 10.000 XP
- Con tabla: nivel 10 = 1.500 XP

Son **incompatibles**. Un usuario en nivel 10 según la tabla sería nivel ~3 según la fórmula.

**Corrección:**
Usar tabla `LEVEL_XP` en `rpg.py`:

```python
from src.config.constants import LEVEL_XP

new_level = 1
for lvl in sorted(LEVEL_XP.keys()):
    if profile.xp >= LEVEL_XP[lvl]:
        new_level = lvl
```

**Prioridad:** 🟡 MEDIA-ALTA

### OBS-V4-003 — Tests solo testean conexión DB

`tests/test_database.py` solo verifica que SQLAlchemy conecta y puede crear filas.

**No hay tests de:**
- Services
- Lógica RPG
- Sistema de dados

Con el nivel de complejidad del RPG, un bug en `calc_streak()` o monedas es muy fácil de meter sin tests.

---

### OBS-V4-004 — `get_db_connection()` mezcla ORM y raw SQLite

`database.py` expone tanto:
- `get_db()` (SQLAlchemy ORM)
- `get_db_connection()` (sqlite3 raw)

El `dice_system.py` generado usa raw. Los services usan ORM.

**Problema:**
Mezclar los dos sobre la misma DB puede causar problemas de transacciones concurrentes (sobre todo con `check_same_thread=False`).

**Recomendación:**
Migrar todo a ORM. Eliminar `get_db_connection()` cuando dados esté reescrito con SQLAlchemy.

## 💬 OPINIÓN DIRECTA

La V4 es un **salto arquitectónico real** sobre la V3. La estructura `src/core` + `src/services` + `src/config` está bien pensada y es mantenible. Los services de hábitos, tareas y eventos están **listos para conectar a una API**.

El problema principal sigue siendo el mismo que en V3: **el sistema RPG está fragmentado**. Ahora hay tres implementaciones distintas del motor RPG (`src/services/rpg.py`, `rpg_service_new.py`, `gamification_updated.py`), y ninguna está completamente alineada con `constants_new.py`.

Lo más inteligente ahora es **no añadir más código RPG** hasta que la API básica (Fase 2) exista y los services actuales estén conectados a endpoints. Una vez tengas algo que corra en el navegador, el RPG se puede integrar limpiamente en el flujo de fin de día.

**La documentación es el punto más fuerte del proyecto.** Es mejor que muchos proyectos profesionales reales. Eso es tuyo y se nota.
